# /brief

Read the project journal and key files. Synthesize them into `.claude/HANDOFF.md` — a dense, honest briefing document designed to get any developer (or Claude) fully oriented in under 2 minutes.

## Steps

### 1. Find the project root

```bash
git rev-parse --show-toplevel 2>/dev/null || pwd
```

### 2. Gather context

Read the following, in order, skipping any that don't exist:

**Journal (primary source):**
- `.claude/journal.md`

**Project identity:**
- `PRODUCT.md`
- `README.md`
- `CLAUDE.md`

**Current state:**
- `TODOS.md`
- `CHANGELOG.md`
- `package.json` or `Cargo.toml` or `pyproject.toml` (whichever exists — just name and version)

**Architecture signals:**
- Glob `src/**/*.ts` or `src/**/*.tsx` or `lib/**/*.rb` — read file names only, not contents, to understand the shape of the codebase
- Read up to 3 files that seem most central (e.g. `App.tsx`, `src/lib/csv-generator.ts`, main entry point)

**Spec files:**
- Glob `spec/*.md` — read any that exist

Do not read every file. Read enough to produce an accurate briefing.

### 3. Write HANDOFF.md

Write to `{PROJECT_ROOT}/.claude/HANDOFF.md`. Overwrite if it exists.

Use this exact structure:

```markdown
# {Project Name} — Handoff

*Last updated: {YYYY-MM-DD HH:MM}*

---

## What this is

{1-3 sentences. The real user, the real pain, the product insight. Written for a developer who has never seen this project.}

## Current state

{Bullet list. What is actually built and working. What is scaffolded but not complete. What is planned but not started. Be honest — "source files written, Expo scaffold not yet run" is more useful than "in progress".}

## Load-bearing decisions

{Pull every DECISION entry from the journal. These are the choices that shape everything. A developer who violates these will cause problems. List as bullets.}

- **{decision}** — {why, if captured in the journal}

## Active warnings

{Pull every WARNING entry from the journal. Landmines. Things that will bite.}

- **{warning}**

## Discoveries

{Pull DISCOVERY entries. Things learned about the domain, the users, or the codebase that aren't obvious from reading the code.}

## Where to start reading

{Tell a new developer the 2-3 files that are most important to read first, and why. Be specific.}

## Recent log

{Last 10 journal entries verbatim, newest first. Any type.}

---
*Generated by /brief — append entries with /log — re-read with /orient*
```

### 4. Confirm

Tell the user:
```
HANDOFF.md written to .claude/HANDOFF.md
{N} decisions, {N} warnings, {N} discoveries captured
Run /orient at the start of any session to reload this context
```
